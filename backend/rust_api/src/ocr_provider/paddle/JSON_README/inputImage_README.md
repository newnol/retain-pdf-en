# inputImage Description

## Structure and Alignment Relationship
- The top level of `json_full.json` contains three main sections: `layoutParsingResults`, `preprocessedImages`, and `dataInfo`. In this example, `layoutParsingResults` is a page-expanded array where each item contains fields like `prunedResult`, `markdown`, `outputImages`, and `inputImage`. The URL in `inputImage` has the form `.../input_img_{page_number}.jpg`, maintaining strict one-to-one correspondence with `preprocessed_img_{page_number}.jpg` in `preprocessedImages` (both arrays have the same length).
- The `dataInfo.pages` list only retains each page's `width` and `height`, used to ensure that backend-parsed coordinate systems can map back to the canvas sizes described by these `inputImage`/`preprocessedImages`. That is: `layoutParsingResults[i]`, `preprocessedImages[i]`, `dataInfo.pages[i]` all represent the same source page, and `inputImage` is the immediate input used by the layout parser.

## Role of inputImage Across the Pipeline
- **Provider adapter**: In the `document_schema` adapter, `layoutParsingResults` is the main structure, and `inputImage` can help the adapter map parsed layouts one-to-one with original images (can be used to verify pageIndex, dimensions, or calculate display ratios). If page-level tracing is needed (e.g., `page_id` for each box in `layoutParserResult`), this field confirms the "source image" of the current data; after aligning with `dataInfo.pages` `width/height`, it can generate a normalized document consistent with visual preview.
- **Trace / Log chain**: During debugging, we bind `inputImage` and `preprocessedImages` URLs to `trace_point` or `reporting` for subsequent observation of layout results (especially for reproducing anti-OCR misdetection or provider upgrade issues). When the layout parser errors, the provided `inputImage` can also be fed directly to `renderer` or `debug explorer` to confirm whether the input at that time was cropped or scaled.
- **Frontend preview**: Frontend needs to display layout overlays; here it should preferentially use the same `inputImage` as the layout parser (input undergoes provider's standard preprocessing before being sent for analysis). `preprocessedImages` is more suitable for showing the "standardized view" of provider output, but `inputImage` is what the layout parser actually reads; using it as canvas background in the frontend can avoid canvas size inconsistency with detection results.
- **Rerun / Cache**: `inputImage` carries large query strings (including authorization/signatures and timestamps), suggesting it may have an expiration period; but since it identifies the layout parser's actual input, caching strategy should at least cache the URL (or download a local copy) so subsequent reruns don't depend on provider's online resources. When rerunning the pipeline, the provider adapter checking `inputImage` consistency helps confirm whether old and new layout outputs are identical; the cacher can bind `inputImage` with `layoutParsingResults` for convenient regeneration of `normalized_document` or `reporting` without re-calling OCR.

## Retention Strategy Recommendations
1. The array order of `layoutParsingResults.inputImage` and `preprocessedImages` must stay synchronized; any provider adapter or regression tool using one of them must verify `page_count`/`numPages`.
2. `inputImage` links have relatively short validity periods; during archiving, it is recommended to choose one of two options: either download, or write URL + version information to storage (e.g., in `reporting` metadata), ensuring trace/renderer can re-request in the future.
3. `preprocessedImages` can be used to display provider-processed results, but for layout reproduction or debugging, prefer `inputImage` because it is the layout parser's actual input; if both are preserved, the frontend can alternately display "original viewport (`inputImage`)" and "standardized view (`preprocessedImages`)".
4. If the rerun/caching layer wants to reuse old paths, it must confirm that the corresponding `dataInfo.pages` dimensions and `layoutParsingResults` page counts are consistent, avoiding incomplete playback of `preprocessedImages`/`layoutParserResult` due to `inputImage` expiration.

Through the above description, we treat `layoutParsingResults.inputImage` as the layout parser's canonical input, `preprocessedImages` as the provider's normalized preview, and both along with `dataInfo.pages` form a page-aligned triangular relationship, providing consistent visual baselines for adapter, trace, frontend, and rerun/caching.
