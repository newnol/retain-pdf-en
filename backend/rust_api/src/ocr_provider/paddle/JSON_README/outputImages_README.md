# layoutParsingResults[*].outputImages Usage Guide

`layoutParsingResults` in `json_full.json` is the outermost structure for each page's OCR output. After getting this array, the adapter/debug scripts will successively do structured processing on each page's `prunedResult`, make strategy decisions for `markdown`, and ultimately attach some visual auxiliary images to `outputImages`. In the current Paddle provider sample, each page's `outputImages` has only one item:

| Key Name | Content Example | Description |
| --- | --- | --- |
| `layout_det_res` | `https://.../layout_det_res_0.jpg?...` | Layout detection overlay image, draws all block polygons/bboxes from `prunedResult.parsing_res_list` back onto the original image; URL is from Paddle OCR online CDN (with temporary authorization) |

If additional keys are added later (e.g., some `crop_*.jpg` / `summary_vis` type) they will also be placed in this dictionary; naming should continue to follow the `<stage>_<purpose>` semantic level for differentiation. `outputImages` is not a required field, but once present it indicates the provider has produced a meaningful visualization at the current stage, which can be used to help understand segmentation results.

## Adoption Strategy for Various Consumers

### Adapter (schema adapter)
- Recommend using `layout_det_res` as an auxiliary reference for structural validation in `document_schema` regression or fixture checks. This image can quickly confirm whether the `block_bbox`/`polygon_points` recorded in `prunedResult` are consistent with the actual detected layout; especially when the normalized document has extra/missing blocks, cross-referencing this image will help locate issues faster.
- Not recommended to directly write arbitrary image URLs into the normalized document. This type of image is a "debug-level" artifact that does not affect downstream schema fields, but a link can be attached alongside the regression report to facilitate new provider reviews of whether key layouts are missed.

### Debugging Tools (scripts, runtime logs)
- `layout_det_res` is the most direct visual debugging entry point: when reproducing a case, downloading this URL locally shows the layout detection overlay. Recommend printing this URL (or writing it to the summary generated by `reporting.py`) when scripts like `regression_check.py` and `validate_document_schema.py` output summaries, so that when the operator sees problems with the normalized document, they naturally open the corresponding page's visual result.
- Other potential `outputImages` (e.g., future layout crop images) should also only be written to logs/filesystem in debug mode, avoiding retaining large numbers of temporary images in the formal data pipeline.

### Frontend Preview/Diagnostics
- `layout_det_res` is very suitable for a "layout QA" visual panel (e.g., in the debug console, connect original image, detection overlay, and normalized tree). Since the URL has authorization and is large, it should be treated as an optional click item and not auto-fetched in the main flow to prevent frontend from frequently triggering CDN verification during formal runtime.
- If you want to show users "cropped images" or "visual read-only images" in the future, you can add `crop_*`, `vis_fit_res` and other fields in `outputImages`, explaining they are specifically for frontend/reports; still constrain through README that they should only be read on QA/diagnostic pages.

## Field Retention Recommendations

- `layout_det_res`: Retain. Even if not main pipeline data, a URL or local file (e.g., `artifacts.py`'s `layout_det_res_*` directory) should be preserved in the provider attachment/regression report for subsequent visual alignment checks.
- Other `outputImages`: If the field name clearly corresponds to a certain debugging/cropping scenario (e.g., `block_crop_res`, `layout_vis`), you can choose whether to persist as needed; but in principle, as long as they are not used to build the normalized document, they fall into the "debug/visualization" category, enabled on demand and documented in README that they should not be parsed into schema.

## Related Field Notes

- `inputImage`: Each `layoutParsingResults` also provides the original input image (`input_img_N.jpg`); frontend displaying overlays should first load this image, then use `layout_det_res` as the overlay layer.
- `preprocessedImages`: The preprocessed image at the outermost level of the overall JSON (e.g., `preprocessed_img_0.jpg`) is the document before detection, suitable as a reference for troubleshooting model preprocessing effects; it does not belong to `outputImages`, so it is only briefly mentioned in the README.

After documenting these conventions in the README, the adapter/debugging scripts can directly refer back to this file without needing to repeatedly determine which images are available across multiple scripts. This also aligns with the current main approach: documents/scripts/regression all center around unified schema, while visualization images only exist as auxiliary information.
