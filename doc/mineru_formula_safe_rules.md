# MinerU Formula Safe Correction Rules

These rules specifically target `raw latex-ish` text output from formula OCR engines like MinerU / UniMERNet.

The goal is not to "make all formulas look perfect", but to:

- Only make low-risk, structured, explainable corrections
- Avoid global string replacement that could break complex formulas
- Make Typst / mitex rendering more stable

## Background

MinerU's formula output is essentially a LaTeX sequence generated by a formula recognition model, not the original canonical LaTeX from the PDF.

Common characteristics:

- Spaces between control words and braces: `\mathrm { C H }`
- Subscripts and superscripts scattered by spaces: `x _ { i , j }`
- Numbers scattered by spaces: `1 . 2 7`
- Local style commands mixed in: `\bf { g }`

## Allowed Corrections

These rules can be kept because they are low-risk and semantically clear:

- `\mathrm { C H } -> \mathrm{CH}`
- `a _ { b } -> a_{b}`
- `a ^ { b } -> a^{b}`
- `a _ 2 -> a_2`
- `a ^ 2 -> a^2`
- Tightening spaces within numbers: `1 . 2 7 -> 1.27`
- `\textsuperscript{...} -> ^{...}`
- A small number of clear OCR noise fixes, e.g., `\mathrm { e V } -> \mathrm{eV}`

## Prohibited Corrections

These rules are prohibited from using global string replacement:

- Global `"{ -> ("`, `"} -> )"`
- Global `" _ { -> _("`
- Global `" ^ { -> ^("`
- Regex-based bulk deletion of braces
- Direct rewriting of `\frac`, `\sqrt`, `\left...\right` without confirming syntax boundaries
- "Guessing" complex formula structure based on length or character set

## Complex Structure Handling Principles

For the following structures, only structured, balanced-bracket-level processing is allowed:

- `\frac{...}{...}`
- `\sqrt{...}`
- `\left ... \right`
- `x_{i,j}`
- `E_{g}^{dir}`
- `\Delta G_{H^*}`
- `\mathbf{v}_{t+1}`

Principles:

- First tighten the atomic binding of `_` / `^`
- Then recursively process contents within groups
- Do not aggressively rewrite unknown groupings

## Engineering Boundaries

We accept the following facts:

- MinerU output is not final LaTeX
- For complex formulas, it's better to under-correct than over-correct
- If a complex structure is already renderable, prefer keeping it as-is
- When rendering fails, prefer selective sanitize rather than aggressive rewriting on the main path

## Current Strategy

Current code should follow:

- `normalizer.py` only performs whitelist-based low-risk tightening
- `typst_formula_renderer.py` does not do global brace replacement
- Complex script structures are handled via balanced bracket recursion
- All new rules must have regression tests added before deployment
